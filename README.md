## 项目本质

这个项目的核心不是写一本传统售后手册，而是为 **售后服务运营自动化** 做准备。

新能源汽车售后运营正在从“人工经验驱动”走向“数据 + 流程 + AI 协同驱动”。要让 AI 真正参与售后运营，前提不是先上模型，而是先把业务拆成清晰、稳定、可执行、可检查的标准化 SOP。

因此，本项目要做的是：

- 把售后运营里的关键业务流程标准化
- 把门店、区域、主机厂之间的责任边界标准化
- 把指标、异常、原因、动作、Owner 标准化
- 把客户承诺、SLA（Service Level Agreement，服务级别承诺）、升级机制和审计机制标准化
- 把未来可以交给系统、规则引擎和 AI Agent 执行的部分提前结构化

一句话：

> **这不是一本静态说明书，而是一套面向 AI 自动化售后运营的 SOP 底座。**

## 为什么要标准化 SOP

AI 能不能参与售后运营，取决于业务是否足够结构化。

如果流程只存在于老师傅经验里，AI 只能做问答；如果流程被拆成标准 SOP、指标口径、异常规则、升级路径和责任矩阵，AI 才能进一步参与：

- 自动识别异常指标
- 自动判断问题等级
- 自动推荐处理动作
- 自动生成工单检查项
- 自动提醒 SLA 超时风险
- 自动汇总门店/区域运营问题
- 自动沉淀知识库和复盘结论
- 自动辅助审计 SOP 是否执行

本项目的文档结构，就是为了让这些自动化能力未来有清晰的业务输入。

## 适用对象

| 数据 / 产品 / 系统团队 | 设计看板、诊断规则、工单字段和系统闭环 |
| AI Agent / 自动化系统设计者 | 将 SOP、指标、诊断卡、SLA 和升级机制转成规则、Prompt、任务流和自动化工作流 |

## 后续演进方向

下一步不是继续堆内容，而是把现有内容进一步转成可执行资产：

- **SOP 字段化**：把每个 SOP 拆成触发条件、输入、动作、Owner、SLA、输出物、升级条件。
- **指标规则化**：把指标异常转成规则，例如阈值、联动指标、异常等级、推荐动作。
- **诊断卡产品化**：把“是什么、异常是什么意思、为什么会异常、关联指标、怎么解决、谁负责”固化为统一模板。
- **知识库结构化**：把政策、话术、案例、故障、SOP 版本纳入可检索知识库。
- **任务流自动化**：把诊断、分派、提醒、复盘、审计做成可执行工作流。
- **AI Agent 化**：让 AI 负责初步识别、归因建议、SLA 风险提醒、复盘摘要和知识库更新建议。

## 维护规则

后续维护建议遵循：

- **先改 Markdown，再导出 Word / PDF**：保证 GitHub 上能看到完整修改历史。
- **新增内容必须进目录**：避免正文有新增、目录看不到。
- **该表格化的内容必须用表格**：尤其是 SLA、权限、指标、流程、RACI。
- **少写解释，多写动作**：手册应指导执行，不堆概念。
- **英文简称首次出现要注释**：例如 SLA（Service Level Agreement，服务级别承诺）。
- **每次版本更新要写版本说明**：说明新增、删除、合并和重大调整。
