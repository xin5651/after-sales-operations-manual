# 第 10 章 软件与智能化售后 SOP


## 10.0 本章定位与适用边界

为什么单独成章：

新能源汽车软件相关售后问题占比在持续上升，传统燃油车售后体系几乎不覆盖。

软件类故障（OTA 失败、车机黑屏、辅助驾驶投诉、远程诊断假阳性等）与硬件故障的处置链路完全不同：技师无法独立闭环，必须依赖软件团队、产品团队、研发团队的联动。

本章把"软件类售后场景"独立标准化，避免一线门店"无 SOP 可循"导致客户投诉升级。

适用边界：

车机系统、智驾/辅助驾驶、OTA 升级、远程诊断、智能座舱、APP/小程序、订阅功能等智能化相关问题。

不覆盖纯硬件故障（属第 3 章业务层 SOP）和电池软件故障（属第 11 章电池全生命周期）。

## 10.1 OTA 升级失败与异常处置 SOP

定位：当 OTA 升级失败、卡死、变砖或灰度异常时，建立从车端识别到客户安抚的完整闭环。

| 流程步骤 | 关键动作 | Owner | 输出物 / 关闭标准 |
| --- | --- | --- | --- |
| 1. 异常识别 | 车端上报失败码、卡死状态、客户反馈三路汇总至 OTA 运营台账。 | OTA 运营 | 失败车辆清单 + 失败码分布 |
| 2. 风险判定 | 按"失败 / 卡死 / 变砖 / 功能异常"四级分类，判定是否触发暂停推送。 | OTA 运营 + 研发 | 风险定级单 |
| 3. 紧急止血 | 高风险版本暂停扩大灰度；已失败车辆自动生成售后待办。 | OTA 运营 | 暂停推送通知 + 待办分发 |
| 4. 车辆恢复 | 能远程修复的远程修复；不能修复的生成到店工单并通知客户。 | 研发 / 门店 | 车辆功能恢复确认 |
| 5. 客户沟通 | 统一话术告知失败原因、预计恢复时间、补偿方案（如适用）。 | 客服 / 服务顾问 | 客户确认记录 |
| 6. 复盘改善 | 失败 TOP 原因分析，调整灰度规则、回滚流程、版本测试用例。 | 产品 + 研发 + 售后 | 版本复盘报告 |

| 指标 | 建议口径 | 观察频率 | 异常信号 |
| --- | --- | --- | --- |
| OTA 升级成功率 | 成功升级车辆数 ÷ 推送车辆数 | 日 / 版本 | 低于阈值时暂停推送 |
| OTA 失败处置时长 | 从失败上报到车辆恢复的中位数小时 | 版本 | 高于阈值代表售后响应慢 |
| 到店修复占比 | 需到店才能恢复的车辆 ÷ 失败车辆 | 版本 | 高比例代表回滚机制不健全 |
| 客户投诉率 | OTA 相关投诉 ÷ 升级车辆数 | 版本 | 高于历史水平触发版本复盘 |

| 风险点 | 前置控制 | 升级触发 |
| --- | --- | --- |
| 推送后大面积变砖 | 强制灰度（< 5% 起步），自动回滚机制 | 失败率超过阈值或出现变砖即停推 |
| 客户车辆长期无法使用 | 应急借用车 / 上门救援 / 优先到店窗口 | 单台车停驶超过 24h |
| 敏感版本（涉智驾）失败 | 法务 + 公关预审；提前准备口径 | 出现任何安全相关投诉立即上报 |

## 10.2 车机异常（黑屏 / 卡死 / 重启 / 砖机）现场处置 SOP

定位：客户报修车机异常时，门店一线能在不依赖研发的情况下完成基础排查与分类升级。

| 故障表现 | 一线判定动作 | 处置路径 | 升级条件 |
| --- | --- | --- | --- |
| 黑屏 / 无显示 | 长按电源 30 秒强制重启；检查保险与车身电压；尝试进入工程模式 | 可恢复 → 记录工单；不可恢复 → 升级远程诊断 | 重启 3 次仍无效 |
| 卡死 / 触屏失灵 | 强制重启；检查最近是否升级 OTA；尝试恢复出厂 | 可恢复 → 观察 7 天；不可恢复 → 主机更换工单 | 7 天内复发 ≥ 2 次 |
| 频繁重启 | 导出日志包；检查电池电压与 ACC 信号 | 日志包发研发；建立观察工单 | 重启频次 > 5 次/天 |
| 变砖（OTA 中断导致） | 不允许门店自行刷机；统一走研发刷机通道 | 车辆封存 + 等待研发刷机工具下发 | 出现即升级研发 |

| 指标 | 建议口径 | 观察频率 | 异常信号 |
| --- | --- | --- | --- |
| 车机故障一次解决率 | 一次到店即解决 ÷ 车机故障工单 | 月 | 低于阈值说明诊断或刷机能力不足 |
| 车机更换率 | 更换主机的车辆 ÷ 车机故障工单 | 月 | 过高需追研发与供应商 |
| 车机故障复发率 | 同车 90 天内复发 ÷ 已修车机故障 | 月 | 高复发说明根因未关闭 |

## 10.3 辅助驾驶 / 智驾投诉处置 SOP（涉法律红线）

定位：辅助驾驶相关投诉处置必须法律合规、不越权承诺、不替研发背锅，同时给客户清晰可追溯的处理路径。

红线：任何门店、客服、销售人员不得向客户口头承诺"自动驾驶"、"可放手"、"接管功能性能等同 L3/L4"等表述。所有功能边界以官方说明书与车机系统提示为准。

| 流程步骤 | 关键动作 | Owner | 输出物 / 关闭标准 |
| --- | --- | --- | --- |
| 1. 受理与定性 | 区分功能投诉 / 体验投诉 / 事故关联投诉三类。事故关联立即升级法务。 | 客服 / 服务经理 | 投诉定性单 |
| 2. 证据采集 | 拉取故障时段行车数据、CAN 日志、ADAS 摄像头记录、客户陈述。 | 远程诊断 + 门店 | 完整证据包 |
| 3. 技术分析 | 由智驾技术团队判定是功能预期内、参数标定问题、还是软件 Bug。 | 智驾团队 | 技术结论书 |
| 4. 客户沟通 | 严格使用法务审核过的统一口径；不得书面承诺超出说明书的功能边界。 | 客服 + 法务 | 客户沟通纪要 + 客户签字确认 |
| 5. 闭环动作 | 功能误解 → 教育 + APP 指引；标定问题 → 召回 / 服务活动；软件 Bug → OTA 排期。 | 产品 + 售后 | 处理动作清单 |
| 6. 监管 / 公关风险预判 | 涉及人身/财产损失的，48 小时内同步法务、PR 与监管事务。 | 法务 / PR | 风险预判报告 |

| 指标 | 建议口径 | 观察频率 | 异常信号 |
| --- | --- | --- | --- |
| 智驾投诉量 | 月度智驾相关投诉总量 | 月 | 环比上涨 ≥ 30% 触发专项 |
| 事故关联投诉占比 | 事故关联智驾投诉 ÷ 智驾投诉总量 | 月 | 上涨即升级法务 + PR |
| 口径合规率 | 客服 / 门店沟通记录抽检合规比例 | 月 | 低于阈值需立即培训整改 |
| 法务介入时效 | 事故关联投诉 48h 内完成法务介入比例 | 月 | 低于 100% 直接复盘 |

## 10.4 远程诊断假阳性 / 假阴性治理 SOP

定位：远程诊断如果告警准确率低，会让一线信任崩塌（"系统又乱报"），最终演变为"不看告警"。本节建立诊断模型的售后侧质量闭环。

| 流程步骤 | 关键动作 | Owner | 输出物 |
| --- | --- | --- | --- |
| 1. 告警分级 | 远程诊断告警按"立即停驶 / 限制行驶 / 提示保养 / 仅提示"四级分发 | 远程诊断 + 售后 | 分级告警清单 |
| 2. 现场验证 | 门店进店后做对照检查；技师反馈"告警准确 / 误报 / 漏报" | 技师 | 验证记录 |
| 3. 误报追因 | 误报 TOP 模型回归至算法团队；持续 2 周高误报模型暂停推送 | 算法 + 售后 | 误报根因报告 |
| 4. 漏报追因 | 客户报修但远程诊断未告警的工单，回流至算法团队补训 | 算法 + 质量 | 漏报清单 |
| 5. 模型迭代 | 按季度评审告警准确率、客户感知、门店反馈，决定模型上 / 下线 | 算法 + 售后运营 | 模型版本记录 |

| 指标 | 建议口径 | 观察频率 | 异常信号 |
| --- | --- | --- | --- |
| 告警准确率 | 现场验证准确告警 ÷ 已验证告警总数 | 周 / 模型 | < 阈值 → 暂停推送 |
| 漏报率 | 客户报修中远程诊断未告警比例 | 月 | 上升说明模型覆盖不足 |
| 一线告警采纳率 | 一线按告警动作执行的比例 | 月 | 低采纳说明信任度下降 |

## 10.5 软件功能减配 / 订阅功能争议处置 SOP

定位：客户对"购车时承诺的功能后期不再提供"、"订阅功能扣费争议"等投诉，需要门店和客服有统一的判定与处置口径。

| 争议类型 | 判定要点 | 处置动作 | 红线 |
| --- | --- | --- | --- |
| 功能延迟交付 | 核对销售合同 / 官方宣发 / 车机功能清单 | 提供 ETA；如长期不能交付提供补偿或退订方案 | 不得回避或推诿研发节奏 |
| 功能减配 / 移除 | 查证是否经过用户授权同意；是否影响行车安全 | 若影响安全 → 召回；若仅体验 → 补偿或恢复 | 不可强制移除安全相关功能 |
| 订阅扣费争议 | 查证扣费记录、订阅协议、退订路径 | 违规扣费立即退款；流程争议走客服升级 | 不得设计反人类退订流程 |
| 硬件预埋功能解锁 | 核对销售承诺与官方政策 | 按官方政策执行；存在销售承诺不一致需追责销售 | 不得让客户为销售承诺买单 |

## 10.6 软件团队 ↔ 售后工单回流机制

定位：现场发现的软件问题必须有结构化通道反馈到研发；研发的版本变更也必须有标准通道下发到一线。两边不能各做各的。

| 方向 | 场景 | 通道 / 频率 | SLA |
| --- | --- | --- | --- |
| 售后 → 研发 | 现场发现的可复现软件 Bug | 研发工单系统，每日同步 | 24h 内分派；P0 当日响应 |
| 售后 → 研发 | 远程诊断误报 / 漏报 | 算法团队周会清单 | 48h 内分析 |
| 售后 → 研发 | 客户投诉中暴露的功能设计缺陷 | 产品月会 VOC 输入 | 月度评审纳入排期 |
| 研发 → 售后 | 版本发布说明 / 已知问题 | 版本简报，发布前 48h | 一线确认接收 + 培训完成 |
| 研发 → 售后 | 新功能上线培训 | 新功能发布前 1 周 | 一线考核通过率 ≥ 90% |
| 研发 → 售后 | 紧急回滚 / 服务活动 | 应急通讯组立即触达 | 4h 内完成一线通知 |

## 10.7 软件售后专项风险红线

🔴 不允许门店私自刷机：所有刷机必须通过研发授权工具与流程，私自刷机产生的损失由门店承担。

🔴 不允许跨车型套用诊断方案：不同车型 / 不同年款的软硬件差异可能很大。

🔴 智驾相关投诉禁止口头承诺：所有承诺以法务审核过的书面口径为准。

🔴 OTA 失败必须有官方告知：不能让客户自行猜测原因；尤其涉及安全功能的版本。

🟡 远程诊断告警必须有人看：不允许"告警了但没人响应"持续超过 SLA。

🟡 客户数据导出需授权：调取行车数据 / CAN 日志 / 摄像头数据，必须有客户书面授权（合规条款详见第 8.4 节）。

## 10.8 本章核心指标合集

本章合计新增 9 项核心指标（编号 S-12 ~ S-20），建议纳入第 5 章指标体系总表统一管理。

| 编号 | 指标名称 | 所属 SOP | 建议目标 |
| --- | --- | --- | --- |
| S-12 | OTA 升级成功率 | 10.1 | ≥ 95%；低于阈值暂停推送 |
| S-13 | OTA 失败处置时长 | 10.1 | 中位数 ≤ 72h |
| S-14 | 车机故障一次解决率 | 10.2 | ≥ 80% |
| S-15 | 车机故障复发率 | 10.2 | < 5% |
| S-16 | 智驾投诉量 | 10.3 | 环比稳定；事故关联立即升级 |
| S-17 | 智驾沟通口径合规率 | 10.3 | ≥ 95%；抽检不合格立即培训 |
| S-18 | 远程诊断告警准确率 | 10.4 | ≥ 90% |
| S-19 | 一线告警采纳率 | 10.4 | ≥ 85% |
| S-20 | 研发→售后版本简报接收率 | 10.6 | 100%（每个版本） |

## 10.9 本章落地优先级建议

| 优先级 | SOP 模块 | 理由 |
| --- | --- | --- |
| P0 | 10.3 智驾投诉处置 | 法律红线，错一次代价极大 |
| P0 | 10.1 OTA 异常处置 | 高频，且容易演变为客户群体性事件 |
| P1 | 10.2 车机异常处置 | 高频，但单台影响有限 |
| P1 | 10.4 远程诊断治理 | 不做会导致整个智能化诊断体系信任崩塌 |
| P2 | 10.5 / 11.6 | 体系化补强，依赖产品 / 研发 / 法务多方协同 |

## 10.10 本章落地评审清单

以下清单用于下一轮版本更新、落地前评审和管理层复盘。它不是草稿问题，而是正式的“待确认/待深化”事项池。

1. 智驾投诉的红线、统一口径和样例话术是否需要进一步细化，并经法务审核后沉淀为客服知识库条目。

2. 远程诊断治理是否需要拆出独立诊断引擎章节，或继续保留在软件与智能化售后章节内。

3. 售后 ↔ 研发回流的 SLA 是否需要按 P0 / P1 / P2 事件等级进一步细分。

4. S-12 ~ S-20 软件类新增指标是否并入第 5 章指标体系总表，或保留为软件专项子看板。

5. 是否新增 T-11《软件类客户沟通话术模板》，覆盖 OTA 失败、车机异常、智驾投诉、订阅争议等场景。
