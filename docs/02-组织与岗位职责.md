# 第 2 章 组织与岗位职责


本章定义售后运营体系的组织分工和关键岗位职责。目的是回答两个问题：谁在干什么、出问题通常找谁对接。岗位名称可按企业实际情况调整，职责边界和升级路径通常保持相对稳定。

## 2.1 组织框架

售后运营采用“总部 — 区域 — 门店”三层管理，在总部内部按业务层与运营层并行设置专业团队。

| 层级 | 主体 | 核心职责 |
| --- | --- | --- |
| 总部 | 售后运营中心 | 策略制定、SOP 与诊断引擎建设、指标体系治理、跨区域调度。 |
| 总部专业线 | 业务层（维保 / 三电 / 充电 / 救援 / OTA / 增值 / 客服） | 对应业务的标准制定、专项治理、跨区赋能。 |
| 总部专业线 | 运营层（网络 / 工单 / 配件 / 技师 / 客户 / 财务 / 质量 / 数据 / 体验 / 合规） | 支撑业务稳定运转的横向能力建设与考核。 |
| 区域 | 区域售后运营 + 区域督导 | 区域内门店指标管理、专项落地、跨店协调、督导稽核。 |
| 门店 | 服务经理 + 服务顾问 + 技师 + 配件员 + 质检员 | 现场作业、客户沟通、SOP 执行、数据留痕。 |

## 2.2 关键岗位职责

以下按总部、区域、门店三个层级，列出关键岗位及其核心职责。岗位名称可按企业实际情况调整，但职责边界通常不建议改变。

总部岗位

| 岗位 | 核心职责 |
| --- | --- |
| 售后运营总监 | 全网售后运营业绩、客户满意度、合规与成本的最终责任人。 |
| 业务运营负责人 | 维保 / 三电 / 充电 / 救援 / OTA / 增值 / 客服 7 大业务专项治理。 |
| 运营体系负责人 | 网络 / 工单 / 配件 / 技师 / 客户 / 财务 / 质量 / 数据 / 体验 / 合规 10 大支撑模块。 |
| 质量运营 | 质量回路（QRQC）、返修与召回、研发联动。 |
| 数据运营 | 车云数据接入、远程诊断、故障预测、规则引擎数据基座。 |
| 体验运营 | NPS / CSI / CES 治理、客户旅程优化。 |
| 风险与合规 | 三包、安全、数据合规、舆情与法务工单。 |
| 系统产品 | 工单系统、客户系统、诊断引擎产品化与规则配置。 |

区域岗位

| 岗位 | 核心职责 |
| --- | --- |
| 区域售后运营经理 | 区域内门店指标管理、专项推进、月度复盘、升级处置。 |
| 区域督导 | 门店稽核、暗访、SOP 执行偏差治理、整改跟踪。 |
| 区域技术支持 | 复杂故障支援、三电 / OTA 专题、技师赋能。 |
| 区域配件主管 | 区域库存与调拨、关键件与大件保障、缺件 TOP N 跟进。 |
| 区域客户关系 | 区域 A 类客诉、媒体 / 监管投诉、跨店协调。 |

门店岗位

| 岗位 | 核心职责 |
| --- | --- |
| 服务经理 | 门店运营第一责任人，对店内 SLA、产值、客诉、质量负总责。 |
| 服务顾问 | 客户接待、需求确认、报价沟通、进度通报、回访闭环。 |
| 技师主管 | 派工、技术指导、新车型上手、师徒制。 |
| 技师 / 高压技师 | 诊断与维修执行、SOP 落地、影像与数据留痕。 |
| 配件员 | 配件申领、入库出库、账实一致、应急调拨。 |
| 质检员 | 完工检验、关键节点影像、不合格件回收检测。 |
| 客服 / 前台 | 预约、咨询、来电分流、初步问题分类。 |

## 2.3 RACI（责任 / 协同 / 咨询 / 知会）速查

R=负责执行，A=最终负责，C=被咨询，I=被知会。本节给出最常用的 8 类典型事件的角色对照，详细分工见各模块 SOP。

| 典型事件 | R 负责 | A 最终负责 | C 协同 | I 知会 |
| --- | --- | --- | --- | --- |
| 客户预约到店维修 | 服务顾问 | 门店服务经理 | 技师主管 / 配件员 | 客户关系 |
| 三电类复杂故障 | 高压技师 + 三电技术支持 | 门店服务经理 | 区域技术支持 / 质量运营 | 数据运营 / 售后总监 |
| A 类客诉 | 客户关系中心 | 区域售后运营经理 | 门店服务经理 / 法务 / 公关 | 售后总监 / 高层 |
| OTA 升级失败 | OTA 产品 + 售后技术支持 | 软件运营 | 客服 / 门店 | 用户 / 区域售后 |
| 关键件缺货 TOP N | 区域配件主管 | 配件运营负责人 | 供应链计划 / 门店 | 售后总监 |
| 质量回路（QRQC） | 质量运营 | 售后运营总监 | 研发 / 供应商 / 售后技术支持 | 区域售后 / 门店 |
| 重大投诉舆情 | 公关 + 客户关系 | 售后运营总监 | 法务 / 区域售后 | 高层 / 监管对接人 |
| 新车型上市售后准备 | 区域售后 + 技师运营 | 业务运营负责人 | 三电 / 配件 / OTA / 体验 | 门店 / 客服 |

## 2.4 升级机制

出现指标异常、客诉升级、合规风险时，按以下顺序升级。每一级必须在规定时长内给出明确动作或继续升级，不允许停留。

| 级别 | 触发条件 | 处置时长 | 决策人 |
| --- | --- | --- | --- |
| L1 门店 | 单店单事件、未达 SLA、未升级客诉 | ≤ 4 小时给出动作 | 门店服务经理 |
| L2 区域 | 门店连续两期未达标、A 类客诉、关键件区域缺货 | ≤ 24 小时给出方案 | 区域售后运营经理 |
| L3 总部专业线 | 跨区域共性问题、批次质量、OTA 重大失败 | ≤ 48 小时给出方案 | 对应业务 / 运营负责人 |
| L4 总监 | 媒体 / 监管 / 法务介入、重大安全事件、品牌风险 | ≤ 12 小时启动，2 小时内首响 | 售后运营总监 |

## 2.5 组织协同原则

一事一 Owner：行业惯例上，指标异常、客诉、专项治理都建议指定唯一 Owner，避免“责任在大家”。

数据先行：决策依据先看指标和样本，再听汇报，避免“嗓门治理”。

向下穿透：总部 / 区域问题建议穿透到门店、车型、批次、人员，避免停在“全网某指标下降”这类汇报口径。

向上反馈：一线发现的共性问题、规则漏洞、研发隐患，建议在 5 个工作日内反馈到对应总部线，否则通常会被视为流程偏差并复盘。

闭环优先：所有问题以“客户确认 + 指标下周期改善”为关闭标准，应避免假闭环、纸面关单。
