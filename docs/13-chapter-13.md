# 第 13 章 售后变更与发布管理 SOP


## 13.0 本章定位与适用边界

为什么单独成章：

售后业务里最常见的"自家踩雷"事件不是流程问题，而是变更没管好：政策刚改门店不知道、工时调价系统还没更新、新车型上市售后没准备好、看板口径变了汇报数字打架等等。

主手册第 6 章诊断引擎产品化路径提到"灰度发布"，但全局变更管理（policy / pricing / training / system / vehicle launch）没有统一流程。

本章建立"变更分级 → 影响评估 → 灰度发布 → 全网生效 → 回退机制"的标准化通道，让售后体系"改而不乱"。

适用边界：

覆盖：服务政策变更 / 工时定额调整 / 配件价格调整 / 系统看板上线 / 新车型上市售后准备 / 应急召回 / 模板表单更新。

不覆盖：研发侧的 OTA 版本管理（属第 11.1 节）、电池供应商切换的产品级管理（属研发与采购）。

## 13.1 售后变更分级与统一通道

定位：所有售后变更必须按"影响范围 + 紧急程度"分级，进入统一变更管理通道，不允许"哪个部门发个邮件就改"。

| 等级 | 判定条件 | 典型示例 | 审批权限 | 生效要求 |
| --- | --- | --- | --- | --- |
| L1 全网级 | 影响全国所有门店或客户 | 保修政策修订、三包口径变更、全网价格调整 | 售后总经理 + 财务/法务 + 总经理 | 灰度 ≥ 14 天 + 全网培训 + 知会客户 |
| L2 区域级 | 影响 1 个或多个区域 / 门店类型 | 区域促销政策、区域结算价调整、网络分级调整 | 售后总经理 + 区域总监 | 灰度 ≥ 7 天 + 区域培训 |
| L3 流程级 | 影响单一业务流程或工具 | SOP 修订、模板更新、看板口径变更、系统功能小升级 | 售后运营中心 + 业务 Owner | 灰度 ≥ 3 天 + 业务方培训 |
| L4 应急级 | 紧急召回 / 法规变更 / 重大事件响应 | 工信部召回、政策强制变更 | 总经理临时授权 | 24h 内全网生效；事后补流程 |

统一变更通道（CRB - Change Review Board）：

所有 L1-L3 变更进入售后 CRB 周会评审。

L4 应急变更允许先执行后补 CRB 备案，但必须在 7 个工作日内补齐。

CRB 由售后运营中心牵头，固定参会方：业务 Owner / 财务 / 法务 / IT / 培训 / 区域代表。

## 13.2 变更影响评估 SOP（六维评估）

定位：变更上 CRB 前，发起方必须填写六维影响评估表，避免"拍脑袋决定、一线背锅"。

| 评估维度 | 关键问题 | 负责评估方 |
| --- | --- | --- |
| 客户影响 | 客户是否需要被告知？告知方式？是否产生新争议？ | 客户运营 + 法务 |
| 财务影响 | 收入 / 成本 / 毛利变化？是否需要预算调整？ | 财务 |
| 运营影响 | 一线动作变化？培训需求？工时变化？ | 售后运营中心 + 区域 |
| 系统影响 | DMS / 看板 / 工单系统是否需要改造？开发工时？上线节点？ | IT |
| 法务/合规影响 | 是否符合三包、召回、数据合规、消费者权益保护？ | 法务 |
| 风险影响 | 失败回退方案？最坏情况预估？是否需要应急储备？ | 业务 Owner + 售后运营中心 |

评估表强制要求：

六维全部填写，不允许"不适用"作为唯一回答。

任何一维评估"高风险"，CRB 需指定专项评审。

评估表归档保留 ≥ 3 年，用于事后追溯。

## 13.3 变更灰度发布 SOP

定位：除 L4 应急变更外，所有变更必须先小范围灰度验证，再全网铺开。"一拍脑袋全网上线"是事故的常见根因。

| 灰度阶段 | 范围 | 观察重点 | 通过条件 |
| --- | --- | --- | --- |
| 1. 内部试点 | 总部 / 售后运营中心内部模拟执行 | 流程是否能跑通、文档是否清晰 | 内部演练无重大障碍 |
| 2. 小灰度 | 1-3 家试点门店（建议覆盖不同店型） | 一线反馈 / 客户反应 / 异常工单 | 试点门店执行率 ≥ 90%，无重大异常 |
| 3. 中灰度 | 1 个区域 / 单一店型全量 | 区域级数据指标变化、客诉趋势 | 关键指标平稳，客诉无异常上升 |
| 4. 全网生效 | 全国所有相关门店 / 系统 | 全网执行率、客户满意度、财务指标 | 稳定 30 天后视为完成 |

灰度门禁（每阶段升级前必须通过）：

🚫 红灯：关键指标异常下降（如 NPS / 一次修复率 / 收入），立即暂停或回退。

🟡 黄灯：出现非预期异常但可控，需 CRB 评估后决定继续灰度还是优化。

🟢 绿灯：所有评估指标正常，按计划进入下一阶段。

| 指标 | 建议口径 | 观察频率 | 异常信号 |
| --- | --- | --- | --- |
| 灰度通过率 | 正常通过全阶段灰度的变更 ÷ 总变更 | 季 | 低于阈值说明评估不严谨 |
| 灰度阶段平均时长 | 从内部试点到全网生效的中位天数 | 季 | 过短代表灰度走形式 |
| 全网回退率 | 全网生效后又回退的变更 ÷ 全网生效变更 | 季 | > 5% 说明灰度机制失效 |

## 13.4 变更全网生效与通知 SOP

定位：变更全网生效不是"在系统里打勾"就完事，必须确保所有相关方"知道、会做、有支持"。

| 通知对象 | 通知形式 | 时效要求 | 签收验证 |
| --- | --- | --- | --- |
| 一线门店 / 区域 | 正式发文 + 培训 + 知识库更新 | 生效前 ≥ 3 天 | 门店签收 + 培训考核 |
| 客服 / 呼叫中心 | 统一话术 + 知识库更新 + 内部培训 | 生效前 ≥ 3 天 | 抽检话术合规率 |
| 客户 / 公众 | 官网 / APP / 短信 / 客服话术 | 生效前 ≥ 7 天（L1 级） | 公示截图归档 |
| 销售 / 市场 | 联席会议 + 内部通报 | 生效前 ≥ 3 天 | 会议纪要签字 |
| 法务 / 监管 | 法务备案 + 监管报备（如需） | 按法规要求 | 备案凭证 |
| 供应商 / 外包 | 协议变更通知 | 生效前 ≥ 7 天 | 回执确认 |

通知三原则：

一致：所有渠道、所有对象的版本完全一致，不允许"邮件版本"和"公告版本"打架。

可追溯：通知发出时间、接收方、签收状态全部留痕。

可培训：复杂变更必须配套培训材料和 FAQ，不能只发文件让一线自己看。

## 13.5 变更回退机制 SOP（必备预案）

定位：任何变更上线前必须有"如何回退"的预案。没有回退方案的变更不允许进入灰度发布。

⚠️ 强制要求：每个变更的 CRB 评估表必须包含"回退方案"章节，否则 CRB 不允许通过。

| 回退场景 | 触发条件 | 回退动作 | 客户沟通 |
| --- | --- | --- | --- |
| 局部回退 | 某区域 / 某门店出现重大异常 | 该范围立即回滚到上一版本 | 该范围客户单独沟通 |
| 全网回退 | 关键指标全网级红灯，影响业务连续性 | 全网回滚 + 应急预案启动 | 全网公告 + 客服话术统一 |
| 软件类回退 | 系统功能上线后出现严重 Bug | 版本回退 + 数据修复 | 受影响客户主动联系 |
| 政策类回退 | 政策变更后引发监管或舆情风险 | 政策暂停 + 法务声明 + 媒体口径 | 法务 + PR 联合应对 |

回退后必做动作：

24h 内召开复盘会议，识别失败根因。

更新 CRB 评估表 + 灰度方案，标注本次失败教训。

更新风险案例库，避免下一次类似变更重蹈覆辙。

## 13.6 新车型上市售后准备 SOP（特种变更）

定位：新车型上市是售后变更里复杂度最高的"超级变更"——同时涉及培训、设备、配件、系统、政策、文档六类变更。本节是售后能否"承接得住"新车型的关键。

| 筹备阶段 | 里程碑 | 关键交付物 | 风险点 |
| --- | --- | --- | --- |
| T-180 天 | 需求输入 | 车型规格 / 销量预测 / 上市城市清单 | 研发输入晚导致后续全部延期 |
| T-120 天 | 技术资料下发 | 维修手册 / 电路图 / DTC 列表 / SDS | 资料不全导致培训和工具准备滞后 |
| T-90 天 | 工具与设备到位 | 专用诊断设备 + 软件 + 工装到位 | 工具采购周期长，需提前锁定供应商 |
| T-60 天 | 配件铺货 | 首批易损件 + 关键件铺货到区域仓 + 首批门店 | 配件 ETA 延误是上市最大风险 |
| T-60 天 | 认证培训 | 首批授权门店技师 + 高压资质培训 + 考核 | 考核通过率 < 90% 不予承修资格 |
| T-30 天 | 系统配置 | DMS 工时定额 + 配件编码 + 索赔规则配置 | 系统配置错会导致索赔失败和投诉 |
| T-30 天 | 客服与话术 | FAQ + 知识库 + 客服培训 | 客服无法应答会立即变成上市负面 |
| T-14 天 | 应急预案 | OTA 失败 / 高频故障 / 客户投诉应急预案演练 | 没有预案 = 出事就乱 |
| T-7 天 | 联席评审 | 售后 + 研发 + 销售 + 市场 + 法务全要素评审 | 联席评审"Go / No-Go"决策 |
| T-0 | 正式承修 | 首批授权门店开放新车型承修 | 首月监控指标，TOP 问题快速响应 |

| 指标 | 建议口径 | 观察频率 | 异常信号 |
| --- | --- | --- | --- |
| 新车型筹备里程碑达成率 | 按时完成节点 ÷ 总节点 | 车型 | < 90% 启动延期评估 |
| 新车型首月一次修复率 | 首月承修工单一次修复 ÷ 总工单 | 车型 | < 历史车型基线 - 5pp 触发专项 |
| 新车型首月客诉率 | 首月客诉 ÷ 首月承修车辆 | 车型 | > 阈值快速响应 |
| 新车型培训覆盖率 | 已完成培训技师 ÷ 应培训技师 | 车型 | < 100% 影响承修能力 |

## 13.7 价格与工时变更 SOP

定位：配件价格、工时定额、内部结算价的调整是高敏感变更——错一个数字可能产生全网级客诉、收入异常、索赔异常。

| 变更类型 | 审批要求 | 系统生效要求 | 客户沟通要求 |
| --- | --- | --- | --- |
| 对客零售工时调整 | CRB L1 + 财务 + 法务 | 所有终端同步生效；不允许新旧并行 | 提前公示 + APP 弹窗 + 客服话术 |
| 配件零售价调整 | CRB L1 + 财务 | DMS / 电商 / 客服系统同步 | 提前公示；存量订单按旧价 |
| 工时定额内部调整 | CRB L2 + 财务 | 系统同步 + 一线培训 | 不对客；门店内部通知 |
| 销售 / 市场内部结算价 | 财务 + 销售 + 售后联签 | 按月生效 | 内部沟通 |
| 保险公司协议价 | 法务 + 财务 + 售后 | 按协议生效日期 | 协议留痕 |

🔴 价格变更系统切换必须"一刀切"：禁止"今天部分门店生效、明天其他门店生效"。

🔴 存量在途订单按旧价执行：避免"客户预约时一个价、到店一个价"的客诉。

🟡 工时调整必须配套测算：调整后的人均产值、毛利率影响必须事前测算。

## 13.8 售后变更与发布专项风险红线

🔴 强制 CRB：所有 L1-L3 变更必须走 CRB 流程，禁止"邮件发个通知就改"。

🔴 强制灰度：除 L4 应急外，禁止跳过灰度直接全网生效。

🔴 强制回退方案：没有回退预案的变更，CRB 不允许通过。

🔴 强制留痕：所有变更评估表、CRB 决议、通知、签收必须归档 ≥ 3 年。

🔴 强制一致：跨渠道、跨对象的通知版本必须一致；禁止"邮件版"和"公告版"打架。

🟡 强制培训：复杂变更必须配套培训和考核，禁止"发个文件让一线自己看"。

🟡 强制复盘：所有 L1-L3 变更生效 30 天后必须复盘；失败回退必须 24h 内复盘。

## 13.11 本章落地评审清单

以下清单用于下一轮版本更新、落地前评审和管理层复盘。它不是草稿问题，而是正式的“待确认/待深化”事项池。

1. L1 / L2 / L3 / L4 变更分级是否符合实际组织规模；大型主机厂、区域集团、单店经销商应有不同复杂度。

2. 六维影响评估是否需要补充标准评估表模板，纳入附录模板体系。

3. 新车型上市售后准备的 T-180 ~ T-0 里程碑是否需要按纯电、增程、混动、换电等车型类型差异化。

4. 价格与工时变更是否需要补充客户沟通话术模板，避免预约价、到店价、系统价不一致引发客诉。

5. S-41 ~ S-46 变更类新增指标是否并入第 5 章指标体系总表，或保留为变更管理子看板。
